### Context (from oneshot-F028 agent)

The previous agent (oneshot-F028-2026-02-13) correctly identified that values layouts differ:
- spark-4.1: `core.postgresql`, `core.hiveMetastore`
- spark-3.5: `core.postgresql`, `core.hiveMetastore` (same!) BUT also has legacy `hiveMetastore` at root
- spark-base: `postgresql` (flat, different path)

**Resolution approach:**
1. Standardize on `core.*` layout (already used by both 3.5 and 4.1)
2. Move spark-base to `core.*` namespace (breaking change for spark-base standalone users â€” document in ADR)
3. Templates in spark-base use `{{ .Values.core.postgresql.* }}` path
4. Only difference: default database name â€” use `{{ .Values.core.hiveMetastore.database.name | default "metastore" }}` (no version suffix in template, let values set the name)
5. Legacy `hiveMetastore` root key in spark-3.5 values â€” keep for backward compat, but primary rendering uses `core.hiveMetastore`

### Risks

- **Breaking change** for anyone using `spark-base` standalone with `postgresql.*` path â†’ document migration in ADR
- **Dual hive-metastore** paths in spark-3.5 (`core.hiveMetastore` + `hiveMetastore`) â†’ phase out root-level in future WS
